题目

编写一个算法来判断一个数n是不是快乐数。

快乐数定义为：
对于一个正整数，每一次将该数替换为它每个位置上的数字的平方和。
然后重复这个过程直到这个数变为1，也可能是无限循环但始终变不到1。
如果可以变为1，那么这个数就是快乐数。
如果n是快乐数就返回true；不是则返回false 。

解法：双指针

此题可以拆分成两个模块，一是求各位平方之和，二是循环和跳出循环。
难点在于何时跳出循环，显然是自己正处于一个无限循环中的时候。那如何知道自己是否正在处于一个循环中呢？想到了另一道题环形链表中的弗洛伊德判圈算法。于是采用快慢指针的方法。

需要清楚一点：如果自身永远不能到达1，说明自己必定处在一个无限循环中，注意是无限循环而非无限不循环。有没有可能无限不循环呢（像无理数那样）？答案是不会，因为10以内的整数个数是有限的。

区分：
无限循环但始终变不到1
无限不循环但始终变不到1

内存管理概论

发表于 2021-03-17 分类于 编程珠玑 → 操作系统 阅读次数：

#11 内存管理概论

64KB内存空间即可以满足所有人的内存需要。——比尔·盖茨

11.1 内存管理的现实基础

程序要运行，必须要加载到内存。因为从外部存储媒介上直接执行指令的做法效率很低，且灵活性很差，因此，人们发明了内存储器来讲需要运行的程序先行加载，再自动执行，从而提高效率和灵活性。

理想状态下，程序员或用户对内存的要求是：大容量、高速度、持久性。但程序员面临的物理现实却是一个由缓存、主存、磁盘、磁带等组成的复杂内存架构。其中，缓存的特点是低容量、高速度、高价格；主存的特点是中容量、中速度、中价格；磁盘的特点是大容量、低速度、低成本；磁带则更持久、速度更慢、价格更低。

（内存架构：缓存——主存——磁盘——磁带）

显然这个架构与程序员对内存的要求相去甚远，要以这个存储架构为基础来提供程序员所需的内存抽象，我们需要一个巧妙的内存管理机制。内存管理机制负责对内存架构进行管理，使得程序在内存架构的任何一个层次上的存放对用户都是一样的。用户无需担心自己的程序是存储在缓存、主存、磁盘还是磁带上，反正运行、计算、输出的结果都一样。内存管理实现这种媒介透明的手段就是虚拟内存。虚拟内存就是操作系统提供的一种抽象。

11.2 内存管理的目标

  1. 地址保护：一个程序不能访问另一个程序的地址空间。
  2. 地址独立：程序指令执行的过程是先将程序指令加载到内存，然后一条条指令读出并执行。每条指令执行时需要读取操作数和写入运算结果，要读取操作数，就需要给出操作数所在的内存地址，这个地址不能是物理地址，因为无法事先确定程序将在何种机器上运行，操作系统自然不可能发出对应于某台机器的物理内存地址。因此指令里的地址是程序空间的虚拟地址（即虚拟空间的程序地址）。也即程序发出的地址与具体机器的物理主存地址无关，彼此是独立的。

11.3 虚拟内存的概念

虚拟内存是计算机系统内存管理的一种技术。它使得应用程序认为它拥有连续可用的内存（一个连续完整的地址空间），而实际上，它通常是被分隔成多个物理内存碎片，还有部分暂时存储在外部磁盘存储器上，在需要时才进行数据交换。

11.4 操作系统在内存中的位置

操作系统是为用户提供服务的，在逻辑上处于用户程序之下。将其置于地址空间的下面，符合人们的惯性思维。另外，操作系统处于地址空间下面还有另外一个实际好处：就是在程序执行复位、中断、陷入等操作时，控制移交给操作系统更方便，因为这种构造下的操作系统的起始地址为0，无须另行记录操作系统所处的位置，PC清零就可以了。清零操作对硬件来说非常简单，不需要从总线或寄存器读取任何数据。

#09 死锁

A set of blocked processes each holding a resource and waiting to acquire a resource held by another process in the set.

死锁的发生，归根结底是因为对资源的竞争。

这里所说的资源就是一个程序工作时需要的东西。资源既可以是硬件，如CPU、内存、磁盘，也可以是软件，如锁、信号量。根据资源是否可以抢占分为可抢占资源和不可抢占资源。可抢占资源是可以从持有者手中强行抢夺过来的资源，且不会发生系统运行紊乱；不可抢占资源则不能从持有者手中强行抢夺，如果强行抢夺将造成系统运行错误。

当然，从绝对概念上说，没有什么资源是不可抢占的。因此不可抢占只不过是相对意义上的概念，目标是保证每个程序都能正确运行。如果抢占一个线程所持的资源后，还能找到某种方式让该程序正确运行下去，则该资源就是可抢占的；否则就是不可抢占的。例如，CPU是可抢占资源，打印机，刻录机是不可抢占资源。

9.2 死锁的描述

线程的执行需要资源，他们都要求以某种顺序来使用资源：

  1. 请求资源
  2. 使用资源
  3. 释放资源

如果请求被拒绝了或者没有批准，那就等待。这种等待有两种应对方式：一是阻塞等待；二是立即返回，执行别的事情，等以后再请求。也可以失败退出，终止线程。立即返回没有等待，因此也没有死锁，而阻塞等待就一定有可能发生死锁。如果每个线程都在等待某个资源，则没有任何线程可以推进，从而形成死锁。

到现在可以给死锁下一个正式定义：

如果有一组线程，每个线程都在等待一个事件的发生，而这个事件只能由该组线程里面的另一线程发出，则称这组线程发生了死锁。

这里的事件通常是资源的释放。死锁状态中，没有线程可以被执行，释放资源或被叫醒。其实对于一般发生死锁的代码，发生死锁的可能性远低于不发生死锁的可能。但是在讨论死锁问题时，我们关注的是死锁的可能，而不是死锁的概率。

9.3 死锁的四个必要条件

  1. 资源独占。也称为资源互斥条件，即资源不能共享，在一个时候只能由一个线程使用，它和资源有限是等价的。
  2. 持有等待。即一个线程在请求新的资源时，其已经获得的资源并没有释放，而是继续持有。如果每个线程在请求资源时都没有持有任何资源，自然也不会阻碍到别的线程运行。
  3. 不能抢占。由此可以推断出，凡是可以抢占的资源，均不会成为死锁的原因。
  4. 循环等待。这时死锁的最后一个条件，你等我，我等你，就产生了死锁。如果此时画出资源占用和请求图，会出现一个回路。

9.4 哲学家就餐问题

哲学家每天只做两件事情：思考和吃饭。吃饭的规矩是：哲学家围坐于圆桌边，每个人的左右手两边都放着一根筷子。如果要吃饭，就需要获得左右两边的筷子（不能用一根筷子吃饭）。

我们可以很自然地得到一个算法，对于每一个哲学家，运行下面的算法：

  1. 等待左边的筷子可用，就拿起左边的筷子。
  2. 等待右边的筷子可用，就拿起右边的筷子。
  3. 吃饭。
  4. 放下两根筷子。

显然，如果每个哲学家严格交叉执行，将出现所有哲学家都拿着左边的筷子，都等待右边的筷子，死锁发生。接下来就该讨论死锁的应对。

9.5 死锁的应对

死锁的应对，从较高的境界来看，有两种策略：

  • 允许死锁发生
  • 不允许死锁发生

每个策略又有两个子策略。对于第一种策略，其子策略为：

  1. 假装没看见死锁，不予理睬。
  2. 死锁发生后，想办法予以解决。

对于第二种策略，其子策略为：

  1. 通过仔细检查，避免难题出现。
  2. 将发生死锁的必要条件消除，杜绝死锁的发生。

操作系统应对死锁的策略与人类应对难题的策略一样，也是两大类，四小种。下面将分别予以论述。

9.5.1 无为而治：允许死锁，不予理睬

防止死锁往往比重启100次代价还高，因此对于普通的操作系统，与其花费大量精力防止死锁，不如直接重启。这是商业操作系统的常见思想。

9.5.2 后发制人：允许死锁，予以修复

对于高可靠性系统、实时控制系统，绝对不允许死锁发生。

  1. 检测死锁。如果想纠正死锁，自然要先知道是否已经死锁了。死锁的原因是资源竞争，只要对系统中的资源拥有情况和资源请求情况进行分析，就可以得知是否发生死锁。死锁的一个特点是资源关系图中包含一个回路，但是判断一个有向图是否含有回路并不是一件容易的事。这种检查算法的时间复杂度是$n^3$。为了操作系统的效率，我们不会直接在图上进行操作。一种效率较高的做法是利用矩阵，这种做法需要维护两个矩阵：资源分配矩阵和资源请求矩阵。

  2. 从死锁恢复。检测到死锁发生后，有几种方法可以从死锁状态恢复。首先可以抢占，即将某个线程所占的资源强行拿走，分配给别的线程。当然，被抢占的线程很可能会不能正确运行，然而这就是纠正死锁的代价。另一个选择是更进一步，不只是抢占某个线程的资源，而是杀掉整个线程。最后一个方法是上翻（rollback），即将整个系统翻转到过去的某个状态。

先礼后兵、后发制人的方法实际上根本行不通。检测与恢复两个部分都存在巨大困难。因此只能走另一条路，即不让死锁发生。死锁一旦发生就不予理睬，不要试图检测和修复死锁。

9.5.3 先发制人：不允许死锁，动态避免

动态避免的原则：在每次进行资源分配时，必须经过仔细计算，确保该资源请求批准后系统不会进入死锁或潜在的死锁状态。潜在的死锁状态指的是尚未发生死锁，但接下来的执行将一定产生死锁，这种状态又称为不安全状态，与之相对的状态称为安全状态。

具体来说，安全状态是指从该状态开始，我们能够找到一种资源分配方法和顺序，使得所有线程都能够获得其需要的资源，从而不会产生死锁。而不安全状态则是从该状态开始，无论无何也不能找到一种资源分配方法和顺序来满足所有线程的资源需求。

定义了安全状态后，死锁的动态避免策略就很简单了，就是要防止系统进入不安全状态。缺点是无法准确计算将来的最大资源需求。因此第三种方法也将被抛弃。

9.5.4 斩草除根：不允许死锁，静态防止

上兵伐谋，其次伐交，再下攻城。——孙子

死锁的检测与修复只不过是攻城的下策，死锁的动态避免只能算其次伐交的中策，而死锁的静态防止才是上策。这个策略的中心思想是清除死锁发生的土壤，而死锁发生的土壤就是死锁的4个必要条件。只要消除4个必要条件中的任意一个，死锁就无法发生。接下来将讨论如何消除这些必要条件。

9.6 死锁的4个必要条件的消除

  1. 消除资源独占：将资源无限增加或把所有资源变为共享。

  2. 消除持有等待：一个线程必须一次请求其所需要的所有资源，而不是一般情况下的请求一点资源，做一点事情，到需要下一个资源的时候再请求一点，获得资源后再继续进行推进。这样就不需要持有一些资源，等待另一些资源。但是这样太过浪费，而且很难一开始就知道一个线程所需要的所有资源。

    一种变通的办法是还像以前那样请求资源，但是如果请求的资源被拒绝，就释放原本已经拥有的所有资源。如此，阻塞的线程不占用任何资源，死锁也就不会发生。然而这种做法容易因为一点小的失败就前功尽弃，造成浪费。

  3. 消除不能抢占：允许从一个线程手上将资源抢夺过来。典型的例子有CPU和内存空间。该方法也有局限性，并不是所有资源都可以不付出代价而抢占，例如锁就不能被抢占，如果从一个线程手上将锁抢过来，后果有可能是不堪设想的。

  4. 消除循环等待：出现循环等待是因为线程请求资源的顺序是随机的，如果规定对资源的请求必须有序进行，则死锁不会发生。典型的例子是锁。如果两个线程需要同时使用两把锁，若它们按照顺序进行请求，则死锁得以避免。

9.7 银行家算法：冒险的代价

银行家算法是仿照银行发放贷款时采取的控制方式而设计的一种死锁避免算法。该算法的策略是动态避免死锁，通过对资源的仔细分配以避免死锁。其特点是可以超额批准客户的信用额度，即所有客户的信用额度之和可以超过银行的全部资本，这就是杠杆。
