RTTI 简介

现代 C++ 对象在运行时一般都知道自己的类型，利用这一点可以进行一点内省和检查安全的转换。Run-time type information(RTTI) 是一种允许在程序执行期间确定对象类型的机制，它是类型内省的特化。

In computing, type introspection is the ability of a program to examine the type or properties of an object at runtime.

Introspection should not be confused with reflection, which goes a step further and is the ability for a program to manipulate the values, metadata, properties, and functions of an object at runtime.

在 C++ 最初的设计中，Bjarne Stroustrup 并没有囊括 RTTI，因为他觉得这种机制会被滥用。RTTI 被添加到语言中是因为当时许多类库作者都自己实现了此功能，导致库之间不兼容。因此就需要在语言级别支持运行时类型信息。

在 C++ 里，RTTI 可用于：

  1. 借助动态铸型操作符dynamic_cast<>做安全的类型转换
  2. 使用typeid操作符和std::type_info以在运行时操作类型信息

本文就是按照上面 2 种应用场景的思路来行文的。

RTTI 只适用于至少有一个虚函数的多态类。但是在实践中，这点也算不上是个限制，因为大部分情况下基类总是必须有一个虚拟的析构函数才能使派生类对象在用基类指针中删除时能够进行适当的清理。

有的编译器还提供了开启/禁用 RTTI 的标志。关闭 RTTI 一般能有助于减少应用程序的总大小，如果你的运行环境是内存受限系统时这点蚊子腿就很有用了。

执行安全的类型转换

dynamic_cast操作符用于将一个引用或指针下移到类层次结构中的一个更具体的类型。与static_cast不同，被dynamic_cast的目标必须是一个指向类的指针或引用，且类型安全检查是在运行时而非编译时进行的。如果类型不兼容，处理引用时将抛出一个异常，处理指针时将返回一个空指针。

假设某个函数将 A 类型的对象作为入参，并且希望在传入的对象是 A 的子类 B 的实例时执行一些额外的操作，这可以使用dynamic_cast来完成，如下所示：

#include <array>
#include <iostream>
#include <memory>
#include <typeinfo>

using namespace std;

class A {
   public:
    virtual ~A() = default;

    void MethodSpecificToA() {
        cout << "Method specific for A was invoked" << endl;
    }
};

class B : public A {
   public:
    void MethodSpecificToB() {
        cout << "Method specific for B was invoked" << endl;
    }
};

void MyFunction(A* mya) {
    B* myb = dynamic_cast<B*>(mya);

    if (myb != nullptr)
        myb->MethodSpecificToB();
    else
        cerr << "Object is not B type" << endl;
}

// this case
void MyFunction(A& mya) {
    try {
        cout << "curr param type: " << typeid(mya).name() << '\n';
        // Cast will be successful only for B type objects.
        B& myb = dynamic_cast<B&>(mya);
        myb.MethodSpecificToB();
    } catch (const bad_cast& e) {
        cout << "Exception " << e.what() << " thrown." << endl;
        cout << "Object is not of type B" << endl;
    }
}

// another implementation
void MyFunction(A& mya) {
    if (typeid(mya).hash_code() == typeid(B).hash_code()) {
        // B& myb = static_cast<B&>(mya);  // also safe
        B& myb = dynamic_cast<B&>(mya);
        myb.MethodSpecificToB();
    } else {
        cout << "now object is not of type B" << endl;
    }
}

int main() {
    array<unique_ptr<A>, 3> array_of_a;
    array_of_a[0] = make_unique<B>();
    array_of_a[1] = make_unique<B>();
    array_of_a[2] = make_unique<A>();

    for (int i = 0; i < 3; ++i)
        MyFunction(*array_of_a[i]);
}

Output:

Method specific for B was invoked
Method specific for B was invoked
now object is not of type B

运行时操作类型信息

typeid运算符

typeid(type)
typeid(expression)

typeid操作符用来在运行时确定一个对象的类型。结果是const type_info&，为左值表达式，它指代一个具有静态存储期的多态类型std::type_info或它的某个派生类型的const限定的对象。

如果表达式是标识了某个多态类型对象的表达式，此时对象的真实类型不能由其提供的静态信息决定，typeid对该表达式求值得到指代该表达式动态类型的std::type_info对象。这种情况下typeid的求值可能涉及运行时开销，主要指虚表查找，除此之外其他情况下都能在编译期间解决。因此，在非多态的情况下，如果只需要类信息，使用typeid通常比动态铸型更受欢迎，因为typeid总是一个常数时间的过程。

另外，必须对指针解引用才能获取其动态类型，否则结果将是该指针的type_info而非指针指向的对象的。例如：

#include <iostream>
#include <typeinfo>
using namespace std;

class Person {
   public:
    virtual ~Person() = default;
};

class Employee : public Person {};

int main() {
    Person person;
    Employee employee;
    Person* ptr = &employee;
    Person& ref = employee;

    cout << typeid(person).name() << endl;
    cout << typeid(employee).name() << endl;
    cout << typeid(ptr).name() << endl;
    cout << typeid(*ptr).name() << endl;
    cout << typeid(ref).name() << endl;
}

Output:

6Person
8Employee
P6Person
8Employee
8Employee

就着这个例子多说一句，类型开头的这些数字意思是类型名里有几个字母，如果是指针还会有个字母P，如果在 namespace 里还会有个字母N。但是注意name()的行为不是 C++ 标准规定的，而是交给编译器厂商实现的，所以不要把这个规律往所有编译器上去套，比如 msvc 就不是这个规律。

如果表达式是通过对一个指针应用一元*运算符所得，且该指针是空指针值，则抛出bad_typeid类型的异常。例如：

#include <iostream>
#include <typeinfo>

using namespace std;

class A {
   public:
    virtual ~A();
};

int main() {
    A* a = NULL;

    try {
        cout << typeid(*a).name() << endl;
    } catch (bad_typeid) {
        cout << "Object is NULL" << endl;
    }
}

Output:

Object is NULL

如果表达式不是指针，也不是对对象基类的引用，则结果是代表表达式静态类型的type_info引用。表达式的静态类型是指在编译时已知的表达式的类型。

此外，在确定表达式的静态类型时，尽可能地忽略引用；如果类型和表达式的类型具有 cv 限定，则typeid的结果指代对应的无 cv 限定类型，即typeid(T) == typeid(const T)。例如：

#include <iostream>
#include <typeinfo>

using namespace std;

int main()
{
    cout << std::boolalpha << (typeid(int) == typeid(const int&));
}

Output:

true

typeid常用于确定模板参数的类型。

#include <iostream>
#include <string>
#include <typeinfo>

using namespace std;

template <typename T>
T max_t(T arg1, T arg2) {
    cout << typeid(T).name() << " is compared." << endl;
    return (arg1 > arg2 ? arg1 : arg2);
}

int main() {
    cout << max_t("li", "xing") << endl;
    cout << max_t("li"s, "xing"s) << endl;
    cout << max_t(3.14, 10.0) << endl;
    return 0;
}

Output:

PKc is compared.
xing
NSt7__cxx1112basic_stringIcSt11char_traitsIcESaIcEEE is compared.
xing
d is compared.
10

iso 标准不保证同一类型上的typeid表达式的所有求值都指代同一个std::type_info实例，但是保证这些type_info对象的std::type_info::hash_code和std::type_index都相同。例如：

#include <assert.h>

#include <iostream>
#include <typeindex>
#include <typeinfo>

class A {};

int main() {
    A* a1 = new A;
    A* a2 = new A;

    const std::type_info& ti1 = typeid(*a1);
    const std::type_info& ti2 = typeid(*a2);

    assert(a1 != a2);

    assert(ti1 == ti2);  // may not!

    assert(ti1.hash_code() == ti2.hash_code());
    assert(std::type_index(ti1) == std::type_index(ti2));

    delete a1;
    delete a2;
}

出于这个原因，本文前面动态铸型那个例子中不能直接用 typeid 的结果来判断是否与某个类型是同一个。

type_info类

type_info类描述了由编译器在程序中生成的类型信息。这个类的对象有效地存储了一个指向类型名称的指针。type_info类还存储了一个可以用于比较或排序的哈希。类型的编码规则和排序顺序是没有规定的，在不同的程序中可能有所不同。

type_info类的接口信息如下：

class _LIBCPP_EXCEPTION_ABI type_info
{
  type_info& operator=(const type_info&);
  type_info(const type_info&);

 protected:
    typedef __type_info_implementations::__impl __impl;

    __impl::__type_name_t __type_name;

    _LIBCPP_INLINE_VISIBILITY
    explicit type_info(const char* __n)
      : __type_name(__impl::__string_to_type_name(__n)) {}

public:
    _LIBCPP_AVAILABILITY_TYPEINFO_VTABLE
    virtual ~type_info();

    _LIBCPP_INLINE_VISIBILITY
    const char* name() const _NOEXCEPT
    {
      return __impl::__type_name_to_string(__type_name);
    }

    _LIBCPP_INLINE_VISIBILITY
    bool before(const type_info& __arg) const _NOEXCEPT
    {
      return __impl::__lt(__type_name, __arg.__type_name);
    }

    _LIBCPP_INLINE_VISIBILITY
    size_t hash_code() const _NOEXCEPT
    {
      return __impl::__hash(__type_name);
    }

    _LIBCPP_INLINE_VISIBILITY
    bool operator==(const type_info& __arg) const _NOEXCEPT
    {
      return __impl::__eq(__type_name, __arg.__type_name);
    }

    _LIBCPP_INLINE_VISIBILITY
    bool operator!=(const type_info& __arg) const _NOEXCEPT
    { return !operator==(__arg); }
};

注：clang 实现，非 iso c++标准，有略微差异。

你不能直接实例化type_info类的对象，因为该类只有一个私有的复制构造函数。构造一个type_info对象的唯一方法是使用typeid操作符。由于赋值运算符也是私有的，所以你也不能复制或赋值type_info类的对象。

msvc 下的type_info::name成员函数返回一个const char*，代表该类型的人类可读名称。msvc 还实现了一个type_info::raw_name成员函数，代表该类型的混淆名称。该名称实际上是以混淆的形式存储的以节省空间。因此，这个函数比type_info::name要快，因为它不需要对 name 解除混淆。type_info::raw_name函数返回的字符串在比较操作中可以直接拿来使用，但不是可读的。如果你需要一个人类可读的字符串，请使用type_info::name函数代替。而对于 gnu gcc 或 clang，name 函数返回的字符串本身就是人类不可读的，因此可以直接用于比较，这点从上面贴出的代码中就可以看出，因为__type_name_to_string在 name 函数和 eq 函数中都在用。

这里提到了混淆的 name 和人类可读的 name，顺便再拓展几句。gcc 没有实现 msvc 中的返回可读名称的接口，但是有几种方式仍然可以在 gcc 里获取解除混淆后的 name，这个过程叫 Demangling。可以用 gnu utils 提供的应用程序 c++filt，或者内置的 abi 接口__cxa_demangle。导出的 abi 函数原型如下：

// 3.4 Demangler API
extern char *__cxa_demangle(const char *mangled_name,
                            char *output_buffer,
                            size_t *length, int *status);

用户文档见：abi Namespace Reference

举个栗子 🌰：

#include <cxxabi.h>
#include <exception>
#include <iostream>

struct empty {};

template <typename T, int N> struct bar {};

int main() {
  int status;
  char *realname;

  bar<empty, 17> u;
  const std::type_info &ti = typeid(u);

  realname = abi::__cxa_demangle(ti.name(), 0, 0, &status);
  std::cout << ti.name() << "\t=> "
            << realname << "\t: "
            << status << '\n';
  free(realname);

  return 0;
}

Output:

3barI5emptyLi17EE       => bar<empty, 17>       : 0

栗子来自 gnu gcc libstdc++ manual - Chapter 28. Demangling

type_index类

The class type_index provides a simple wrapper for type_info which can be used as an index type in associative containers (22.4) and in unordered associative containers (22.5). – N4901

type_info本身不能作为关联型容器的索引，所以有了type_index类，本质上是把前者简单包装了一下。它和 type_info 对象的关系是通过一个const type_info*指针来维持的：

class _LIBCPP_TEMPLATE_VIS type_index
{
    const type_info* __t_;
public:
    _LIBCPP_INLINE_VISIBILITY
    type_index(const type_info& __y) _NOEXCEPT : __t_(&__y) {}

    _LIBCPP_INLINE_VISIBILITY
    bool operator==(const type_index& __y) const _NOEXCEPT
        {return *__t_ == *__y.__t_;}
    _LIBCPP_INLINE_VISIBILITY
    bool operator!=(const type_index& __y) const _NOEXCEPT
        {return *__t_ != *__y.__t_;}
    _LIBCPP_INLINE_VISIBILITY
    bool operator< (const type_index& __y) const _NOEXCEPT
        {return  __t_->before(*__y.__t_);}
    _LIBCPP_INLINE_VISIBILITY
    bool operator<=(const type_index& __y) const _NOEXCEPT
        {return !__y.__t_->before(*__t_);}
    _LIBCPP_INLINE_VISIBILITY
    bool operator> (const type_index& __y) const _NOEXCEPT
        {return  __y.__t_->before(*__t_);}
    _LIBCPP_INLINE_VISIBILITY
    bool operator>=(const type_index& __y) const _NOEXCEPT
        {return !__t_->before(*__y.__t_);}

    _LIBCPP_INLINE_VISIBILITY
    size_t hash_code() const _NOEXCEPT {return __t_->hash_code();}
    _LIBCPP_INLINE_VISIBILITY
    const char* name() const _NOEXCEPT {return __t_->name();}
};

与 type_info 的另一个显著差别就是 type_index 是 CopyConstructible 和 CopyAssignable 的。

由于 type_index 上面两个重要特征，它可以用于把类型的 raw_name 和其它某些信息关联起来。因为 C++ 本身不支持反射，所以社区中很多开源反射库广泛借用了这两种特征。

举个例子，比如可以把某种类型的 typeinfo 对象的 raw name 和 human readable name 进行关联：

#include <iostream>
#include <memory>
#include <string>
#include <typeindex>
#include <typeinfo>
#include <unordered_map>

using namespace std;

struct A {
    virtual ~A() {}
};

struct B : A {};
struct C : A {};

int main() {
    unordered_map<type_index, string> timap;

    timap[type_index(typeid(int))] = "int";
    timap[type_index(typeid(double))] = "double";
    timap[type_index(typeid(A))] = "A";
    timap[type_index(typeid(B))] = "B";
    timap[type_index(typeid(C))] = "C";

    int i;
    double d;
    A a;

    // note that we're storing pointer to type A
    unique_ptr<A> b(new B);
    unique_ptr<A> c(new C);

    cout << "i is " << timap[type_index(typeid(i))] << '\n';
    cout << "d is " << timap[type_index(typeid(d))] << '\n';
    cout << "a is " << timap[type_index(typeid(a))] << '\n';
    cout << "*b is " << timap[type_index(typeid(*b))] << '\n';
    cout << "*c is " << timap[type_index(typeid(*c))] << '\n';
}

Output:

i is int
d is double
a is A
*b is B
*c is C

何时使用

Only use RTTI if you have a very good reason to do so and only use it if you know what you are doing.

应该谨慎使用 RTTI。诸如多态和模板等语言机制几乎在所有场景下总是优于 RTTI。虽然凡事都有例外，但 RTTI 现在的立场跟 goto 差不多尴尬，rtti 并非通向正确且健壮的设计的捷径。相比于反射库这种基础库，业务代码更不太可能用 rtti，以至于可以把业务逻辑中出现的 rtti 统统视为坏味道，实际上如果出现 rtti 也确实意味着 liskov 替换原则大概率没有被很好地遵守。

Maybe my experience is atypical but RTTI and exceptions are features of C++ that I never see used in real-world C++ almost regardless of the application. When I have seen them used, it is by people new to C++ that haven’t fully figured out the common development idioms of the language yet.
